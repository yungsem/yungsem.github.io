# 分布式锁

# 分布式锁需要具备的特性

为了确保分布式锁可用，必须保证实现的分布式锁满足下面四个特性：

  • 互斥性。在任意时刻，只有一个客户端能持有锁，其他尝试获取锁的客户端都将返回失败或阻塞等待。
  • 健壮性（不会发生死锁）。某一个客户端在持有锁期间崩溃了而没有主动释放锁，其他客户端也能正常获取锁，而不是无休止的等待锁释放出现死锁。
  • 唯一性。加锁和解锁必须是同一个客户端，客户端自己不能把别人加的锁给释放了，自己持有的锁也不能被其他客户端释放。
  • 高可用。不必依赖于全部 redis 节点正常工作，只要大部分的 redis 节点正常运行，客户端就可以进行加锁和解锁操作。

# 基于 redis 单节点的分布式锁算法

# 加锁（获取锁）

加锁的基本思路是客户端向 redis 中存入一个 (k,v) 键值对。k 为锁的名字（下面以 lockName 为例），v 为客户端唯一标识（下面以 clientId 为例）。如果存入成功，就表示加锁成功。加锁需要使用 SET 命令：

SET key value NX PX 30000

NX 选项表示如果 key 不存在，则存入成功，返回 OK 。如果 key 已经存在，则不存入，返回 <nil> 。 PX 选项用于设置 key 的过期时间，单位毫秒。

示例：

127.0.0.1:6379> set mylock client1 NX PX 30000
OK
127.0.0.1:6379> set mylock client1 NX PX 30000
(nil)

借助 SET 命令的 NX 选项，我们可以让锁满足互斥性，即执行上述命令获取锁，如果返回 OK ，则表示获取成功；如果返回 nil ，则表示有其他客户端已经获取锁，该客户端只能等待锁的释放。

借助 PX 选项，我们可以让锁满足健壮性。PX 选项为锁设置了过期时间，如果客户在持有锁期间挂了，过期时间可以保证锁会自动释放，不会发生死锁。

# 解锁（释放锁）

解锁需要满足唯一性，即只能释放自己加的锁，不能把别的客户端加的锁给释放了。

我们在加锁时将 value 设置为客户端唯一标识，为实现唯一性做好了铺垫。

所以解锁其实需要两步：第一步，使用 GET 命令获取 value 值；第二步，判断 value 是否等于自己的 clientId ，如果 value 等于 clientId ，就使用 DEL 命令删除 key ，释放锁。如果 value 不等于 clientId ，则证明自己加的锁已经被释放了，现在是别的客户端持有锁，什么都不需要做。

我们必须保证第一步的 GET 命令和第二步的 DEL 命令原子性执行。因为在并发环境下，我们到 DEL 之前那一时刻，可能有别的客户端正好加锁成功，此时我们再执行 DEL 命令，正好把别的客户端刚加的锁释放掉了，违反了唯一性。

但是两个命令如何保证原子性呢？redis 提供了 lua 脚本，lua 脚本的执行是原子性的，我们可以将这两个命令放到 lua 脚本中，然后发给 redis 执行。

脚本如下：

if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end

# Spring 实践

package com.yungsem.springbootredis.distributedlock;

import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.data.redis.core.script.DefaultRedisScript;
import org.springframework.stereotype.Component;

import javax.annotation.Resource;
import java.util.Collections;
import java.util.concurrent.TimeUnit;

/**
 * @author yangsen
 * @version 1.0
 * @since 2020-08-02
 */
@Component
public class RedisLock {
    private static final Long RELEASE_SUCCESS = 1L;
    @Resource
    private StringRedisTemplate stringRedisTemplate;

    /**
     * 加锁
     *
     * @param lockKey    锁的名称
     * @param clientId   请求加锁的客户端唯一标识
     * @param expireTime 过期时间
     * @return 加锁成功：true 加锁失败：false
     */
    public boolean lock(String lockKey, String clientId, int expireTime) {
        Boolean ok = stringRedisTemplate.opsForValue().setIfAbsent(lockKey, clientId, expireTime, TimeUnit.MINUTES);
        return ok != null && ok;
    }

    /**
     * 解锁
     *
     * @param lockKey  锁的名称
     * @param clientId 请求加锁的客户端唯一标识
     * @return 解锁成功：true 解锁失败：false
     */
    public boolean unlock(String lockKey, String clientId) {
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end";
        // 指定 lua 脚本，并且指定返回值类型
        DefaultRedisScript<Long> redisScript = new DefaultRedisScript<>(script, Long.class);
        // 参数一：redisScript，参数二：key列表，参数三：arg（可多个）
        Long result = stringRedisTemplate.execute(redisScript, Collections.singletonList(lockKey), clientId);
        // 返回
        return RELEASE_SUCCESS.equals(result);
    }
}

# 主从同步时锁丢失问题

在使用哨兵模式的 redis 部署方案下，正常情况下，客户端 A 向主节点获取锁之后，锁会同步给从节点。但如果在锁同步到从节点之前，主节点挂了，在哨兵的帮助下，从节点升为新的主节点，但此时新的主节点上是没有客户端 A 加的锁的。如果这时客户端 B 也来申请锁，是可以申请成功了，这出现了客户端 A 和客户端 B 都加锁成功了，违背了互斥性。

# Redlock

Redlock 是 redis 官方提供的一个分布式锁的方案，主要考虑的是在 redis 集群环境下，如何正确的实现分布式锁。

算法思路如下：

  • client 获取当前时间戳，记为 T0 。
  • client 按顺序依次向所有 redis 节点申请锁。 在这个过程中，要设置获取锁这个请求的超时时间远远小于锁的过期时间，因为有的节点可能已经挂了，client 不能傻等，等的时间都超过了锁的过期时间，那么本轮获取锁就没有任何意义。比如 TTL 是 5 秒，那么超时时间最多 1 秒，超过 1 秒还没获取到锁，就放弃这个节点，转向下个节点获取。
  • client 向所有节点都申请一遍，记录最后一个节点申请返回的时间，记为 T1 。
  • 判断是否至少从 N/2 + 1 个节点那成功获取到了锁，比起 T1 - T0 < TTL ，就认为 client 成功获取到了锁，并且锁的过期时间不是 TTL ，而是 T1 - T0 ，严格点还应减去时钟漂移。
  • 如果 client 获取锁没有满足上面的条件，就认为获取锁失败。此时应将已经从部分节点获取到的锁释放掉，不能影响别的 client 获取锁。
