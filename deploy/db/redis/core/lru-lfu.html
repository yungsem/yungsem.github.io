# 缓存的过期策略

# 过期策略相关配置

maxmemory <bytes>

指定 redis-server 能使用的最大内存大小。如果 redis-server 使用的内存已经到达该值，则会使用 maxmemory-policy 项中配置的过期策略筛选出需要逐出的 key ，然后清除这些 key 以获得空间。如果这一番努力之后仍然没有足够的空间，或者 maxmemory-policy=noeviction ，则禁止客户端再进行写操作（会返回错误），但是允许客户端进行读操作。

maxmemory-policy noeviction

过期策略。上面已经提到，redis-server 空间不足时，会使用该项配置的过期策略筛选出需要逐出内存的 key 。过期策略有如下八种：

  • noeviction： 不删除任何 key 。禁止客户端再进行写操作（会返回错误），但是允许客户端进行读操作。
  • volatile-lru： 在设置了过期时间的 key 中使用 LRU 算法筛选出 key ，删除。
  • allkeys-lru： 在所有的 key 中使用 LRU 算法筛选出 key ，删除。
  • volatile-lfu： 在设置了过期时间的 key 中使用 LFU 算法筛选出 key ，删除。
  • allkeys-lfu： 在所有的 key 中使用 LFU 算法筛选出 key ，删除。
  • volatile-random： 在设置了过期时间的 key 中随机挑选 key ，删除。
  • allkeys-random： 在所有的 key 中随机挑选 key ，删除。
  • volatile-ttl： 挑选最接近过期时间的 key ，也即 TTL 值最小的 key ，删除。

# LRU 算法

LRU 算法也即最近最少使用算法。这个算法的关键就是要知道哪个 key 是最近最少使用的。

redis 为每一个 key 都记录了该 key 的空闲时间 idle time 。idle time 越大的 key ，就是较少被访问的。

redis LRU 算法不是真正的 LRU 算法，而是近似的 LRU 算法。算法的思路大致如下：

  • 第一次，随机选择 5 个 key ，将 5 个 key 都放入候选池中。
  • 后面每一次，随机选择 5 个 key ，将其中 idle time > 候选池中最大的 idle time 的 key ，放入候选池中。
  • 当候选池满了的时候，将池中 idle time 最大的 key 删除。

LRU 算法也有弊端：就是假如一个 key 值在以前都没有被访问到，然而最近一次被访问到了，那么就会认为它是热点数据，不会被淘汰。

然而有些数据以前经常被访问到，只是最近的时间内没有被访问到，这样就导致这些数据很可能被淘汰掉，这样一来就会出现误判而淘汰热点数据。

# LFU 算法

LFU 算法也即最近最频繁使用算法。

redis 会记录所有 key 最近一段时间内被访问的频率。频率越小，表示越不频繁被访问到，就可以删除。

# 删除过期键的策略

在Redis种有三种删除的操作此策略，分别是：

  • 定时删除：创建一个定时器，定时的执行对key的删除操作。
  • 惰性删除：每次只有再访问key的时候，才会检查key的过期时间，若是已经过期了就执行删除。
  • 定期删除：每隔一段时间，就会检查删除掉过期的key。

# RDB 和 AOF 的淘汰处理

在RDB中是以快照的形式获取内存中某一时间点的数据副本，在创建RDB文件的时候可以通过 save 和 bgsave 命令执行创建RDB文件。

这两个命令都不会把过期的 key 保存到 RDB 文件中，这样也能达到删除过期 key 的效果。

当在启动 Redis 载入 RDB 文件的时候，Master 不会把过期的 key 载入，而 Slave 会把过期的 key 载入。

在 AOF 模式下，Redis 提供了 Rewite 的优化措施，执行的命令分别是 REWRITEAOF 和 BGREWRITEAOF ，这两个命令都不会把过期的 key 写入到 AOF 文件中，也能删除过期 key 。
