# Java 内存模型

# Java 内存模型是什么

对于一个 Java 程序，在多线程环境下，下面三种变量可能会出现线程安全问题：

  • 成员变量（实例变量）
  • 类变量
  • 数组对象的元素

注：下文提到的 变量 都是指上述三种变量之一。

为了解决这个问题，Java 定义了一套规则，规定了在多线程环境下，应该怎么访问上面三种变量，以保证线程安全。这一套规则就是 JSR-133 规范，该规范在 JDK 5 中得以成熟和完善。

我们所说的 Java 内存模型（Java Memory Model, JMM）可以理解为 JSR-133 规范的具体内容。

# 主内存和工作内存

JMM 规定变量都必须存储在主内存（Main Memory）中。每一个线程都有一个自己的工作内存（Work Memory）。工作内存存有变量的副本。线程对变量的读写操作都只能在自己的工作内存中完成，然后刷新到主内存中。工作内存为线程私有，线程只能操作自己的工作内存，不能操作别的线程的工作内存。

我们知道，JVM 在运行时将内存分为以下几个部分：

  • 方法区
  • 堆
  • 虚拟机栈
  • 程序计数器
  • 本地方法栈

那么这里所讲的主内存/工作内存和 JVM 运行时数据区域有没有什么对应关系呢？其实，它们之间并没有什么严格的一一对应的关系。但是可以在理解上做一个伪对应，就是主内存对应堆，工作内存对应虚拟机栈。

# 线程对变量的 8 种操作

上面我们说到，JMM 规定变量都存储在主内存中，但是每个线程的工作内存持有变量的副本。如下图所示：

上图中工作内存中的变量 V` 就是主内存中的变量 V 的副本。

JMM 规定了线程对变量可以进行的操作有 8 种。为了表述更加清晰一点，我们使用上面图示中的变量 V 和对应的副本变量 V` 做示例说明：

操作（英文） 操作（中文） 作用对象 说明
lock 锁定 主内存 锁定变量 V 。该变量一旦被锁定，就不能再被其他线程锁定了
unlock 解锁 主内存 解除对变量 V 的锁定。
read 读取 主内存 将变量 V 的值传送到该线程的工作内存的一个临时区域中
load 载入 工作内存 将 read 操作存放在临时区域中的值存入变量 V` 中
use 使用 工作内存 将变量 V` 的值传递给执行引擎
assign 赋值 工作内存 将从执行引擎接收到的值赋值变量 V`
store 存储 工作内存 将变量 V`值传送到主内存中的一个临时区域
write 写入 主内存 将 store 操作存放在临时区域中的值存入变量 V 中

# 8 种操作必须遵守的原则

JMM 规定了线程可以对变量进行 8 种操作，但是为了保证变量在多线程环境下是线程安全的，JMM 又规定了进行这 8 种操作时必须遵守的原则。如下：

  • read 和 load 必须成对出现；store 和 write 也必须成对出现。
  • 线程对变量执行 assign 操作之后，必须执行 store + write 操作把新值同步回主内存中。
  • store + write 操作必须有对应的先行的 assign 操作。如果没有对变量执行 assgin 操作，则不能执行 store + write 操作将值同步回主内存中。
  • 不允许对一个未被初始化的变量执行 use 或 store 操作。 load 或 assign 操作可以对一个变量进行初始化。
  • 一个变量在同一时刻只允许一个线程对其进行 lock 操作。但一个线程可以对同一个变量重复执行多次 lock 操作，此时如果想要解锁的话，必须执行相同次数的 unlock 操作。
  • 如果对一个变量执行 lock 操作，那么该变量在工作内存中副本变量的值就会被清空。也即在工作内存中使用该变量对应的副本变量时，需要重新初始化。
  • 必须先 lock 一个对象，才能对该对象执行 unlock 操作。没有先 lock ，就不能 unlock 。
  • 对一个变量执行 unlock 操作之前， 必须先将该变量的值同步回主内存中，也即要执行 store + write 操作。

# 原子性、可见性和有序性

# 原子性

当我们说某一个对变量的操作是具备原子性的，是指这个操作是不可分割的，要么成功，要么失败。不会成功一部分，失败一部分。对应到多线程环境下，线程 A 对某一个变量进行原子性操作时，不会被线程 B 穿插打断。而是必须等到线程 A 操作完成之后，线程 B 才能来接着操作该变量。

在上面我们讲过的线程对变量的 8 种操作中，以下 6 个操作是原子性的：

  • read
  • load
  • assign
  • use
  • store
  • write

我们看几个 Java 代码的示例：

int a = 10;

该操作是一个原子操作。在 Java 中，对基本类型的访问和读写都是具备原子性。

a++;

该操作就不是一个原子操作。对变量 a 进行自增，其实可以拆分为下面三个操作：

  • 读取变量 a 的值。
  • 对读取到的值加 1 。
  • 将加 1 得到的值赋值给变量 a 。

# 可见性

一个操作具备可见性，是指线程 A 对自己工作内存中的变量 V` 的值进行修改之后，能立即将修改后的值同步回主内存对应的变量 V 中，以保证其他线程在使用变量 V 时，得到的是变量 V 的最新值。

# 有序性

要理解有序性，就必须先理解 指令重排序优化 ，先看下面的代码：

public class Demo {
    // flag 会被多个线程共享
    private boolean flag = false;

    public void demo() {
        // 操作 A 成功执行之后，flag 才能置为 true
        flag = true;

        if (flag) {
            // .....
        }
    }
}

这段代码经过编译和解释之后，会编程一串汇编指令，CPU 为了提高执行效率，可能将 flag = true 对应的汇编指令（可能不止一条）提前到操作 A 之前执行，这就是指令重排序优化。是 CPU 层面的。

在单线程环境下，这种指令重排序优化对程序的执行结果不会产生影响。即无论是重排序，还是不重排序，上面的代码执行结果都是一致的。

但是在多线程环境下，这种优化就有可能导致问题。例如有两个线程，线程 A 正在执行操作 A ，理论上此时 flag 是不能为 true 的，因为操作 A 还没有结束。如果此时有指令重排序优化，将 flag = true 提前执行了，那么线程 B 拿到 flag 的值就是 true ，然后它会进入 if 分支，但是由于操作 A 还没有结束，执行 if 分支里面的代码会报错。

禁止上述指令重排序优化，就是有序性。
