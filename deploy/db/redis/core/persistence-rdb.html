# 持久化：RDB

# 原理

RDB 持久化是一种快照形式的持久化。redis 每隔一段时间（根据配置）将 redis 当前内存中的数据写入到 rdb 文件中。

客户端触发 RDB 持久化主要有两个命令：

  • save
  • bgsave

# save

由于 redis 使用单线程执行客户端发来的命令，当执行 save 命令时，redis 是阻塞住的，不会响应其他的客户端命令，直到 save 命令执行结束，数据写入 rdb 文件完毕。所以 save 命令在生产环境下是不建议使用的，建议使用 bgsave 命令。redis 后台根据配置自动触发 RDB 持久化也是使用 bgsave 命令的方式。

bgsave 命令只在 fork() 调用的时候是阻塞的，这个过程比较短，一旦 fork() 完成，主进程就可以继续响应客户端命令了，子进程负责将数据写入到 rdb 文件中。

fork() 不会将主进程的内存完整的复制一份，而是和主进程共享同一份内存地址空间。那么带来的问题是：子进程在持久化数据的同时，主进程还在继续响应客户端的写命令，那么子进程会把主进程新写入的数据也持久化吗？

解答上面的疑惑，我们需要了解一点 fork() 和 copy on write 机制。

# fork()

fork() 是一个系统调用，用于创建子进程。

子进程共享父进程的内存空间，也即内存数据。但是他们各自对共享内存空间里数据的修改，互相不可见。也就是说，父进程对内存数据的修改，子进程不知道，反之亦然。

这就解答了上面的疑惑，在执行 bgsave 命令时，子进程持久化的数据不包含主进程新写入的数据，只包含 fork() 结束那一时刻内存的数据。

那么操作系统是如果做到让父进程和子进程共享同一内存地址空间但更改数据互不影响的呢？这就要用到 copy on write 机制了。

# copy on write

父进程调用 fork() 创建子进程完成后，系统就将父进程所有内存页的权限设置为 read-only ，然后子进程的内存地址空间指向父进程的。实现内存共享。

当父进程或子进程（下面以父进程为例）往某一个内存页写数据时，CPU 检测到该内存页的权限是 read-only ，无法写入，于是就触发页异常中断（page-fault），陷入对应的内核中断程序。

在内核中断程序中，内核将触发中断的那个内存页复制一份（只复制这一个页）出来，将要写的数据写入，然后将父进程中引用异常页的地址更新为这个新页的地址。

这样一来，父进程成功写入了新数据，只不过新数据写入到了一个新的内存页，但是子进程是不知道这个新内存页的。做到了父进程和子进程新写入数据的互相隔离。

# 优点

  • RDB 持久化是每隔一段时间将数据写入到 rdb 文件中，如果我们按照一定的规则命名 rdb 文件，并且每次持久化都保留原来的 rdb 文件而不是覆盖，则我们很容易选择其中一个版本进行恢复。
  • rdb 文件是一个压缩文件，服务器间传输很方便，有利于数据恢复。
  • RDB 持久化使用子进程完成持久化（磁盘IO）的动作，主进程不用进行磁盘 IO 的操作，这对主进程响应客户端命令的效率是一个很好的保障。
  • 使用 rdb 文件恢复数据比 AOF 快。

# 缺点

  • 由于 RDB 持久化触发有时间间隔，有可能会造成量比较大的数据丢失。
  • fork() 也有一定的性能开销，fork() 执行过程中，主进程是阻塞的。如果内存中数据量比较大的话，fork() 耗时比较长，会导致主进程停止响应客户端命令几秒钟，这在有些场景下比较要命。

# 相关配置

save 60 10000

60 秒内有 10000 个 key 发生变化，就触发 bgsave 。

stop-writes-on-bgsave-error yes

如果最近一次 bgsave 失败，则 redis 停止接收客户端命令。

这样做的好处是尽早暴露问题，因为 bgsave 一直失败，但客户还能继续写入新数据的话，相当于关闭持久化机制。后期如果想恢复数据，发现 rdb 文件是没用的，根本没法恢复。

rdbcompression yes

是否启动压缩。一般都建议开启。

rdbchecksum yes

是否开启 rdb 文件校验。如果开启，在写入或恢复时，都会校验，大约有 10% 的性能消耗。

dbfilename dump.rdb

rdb 文件的名称。

dir ./

rdb 文件存放的目录。

# FLUSHALL 和 SHUTDOWN 命令的问题

FLUSHALL 和 SHUTDOWN 命令一执行，内存中的数据先被全部清空，然后触发 RDB 持久化，生成一个新的 rdb 文件覆盖旧的 rdb 文件，此时新的 rdb 文件中的数据的内存中数据的备份，也就是什么也没有。因为内存中的数据刚刚被 FLUSHALL 命令清空了。
